Safe multisig approvals for protected Git changes

Require a Safe before GitHub says a commit is good to merge.

SafeGit creates an EIP-712 approval request for a Git commit, collects Safe owner signatures, stores shared state in Postgres, and lets CI enforce a Safe Verified required check.

What it is

Not GitHub’s native Verified badge. A stricter required check.

GitHub natively understands GPG, SSH, and S/MIME commit signatures. A Safe is an ERC-1271 smart account, so SafeGit uses the practical route: a GitHub required check named something like Safe Verified.

CLI

Commit-aware requests

Reads repo, branch, commit SHA, tree SHA, author, committer, Safe address, chain ID, nonce, and expiry.

EIP-712

Human-reviewable signing

Owners sign a structured GitCommitApproval payload instead of a bare hash with no context.

CI

Merge enforcement

safegit verify exits non-zero unless the commit is approved, making it fit GitHub branch protection.

Architecture

The boring version that actually works.

SafeGit is a CLI plus API plus Postgres. Local-only state is fine for demos; teams need shared state so signers, CI, and GitHub checks all see the same approval.

Install

Install locally with pnpm only.

SafeGit is intentionally pnpm-only. No npm lockfile, no mixed package managers.

Local CLI install

git clone https://github.com/agent-cortex/safegit.git
cd safegit
pnpm install
pnpm link --global

Run API + Postgres with Docker

cd safegit
docker compose up --build

API defaults to http://127.0.0.1:8787.

Exact flow

Step-by-step: from repo to Safe-approved commit.

This is the complete happy path. Use Sepolia for testing; use the real chain ID and Safe address for production.

Start Postgres and the API

The Docker Compose path is the fastest way to run the shared backend.

cd safegit
docker compose up --build

Export the DB URL for CLI commands

The CLI and server must point at the same database.

export SAFEGIT_DATABASE_URL='postgres://safegit:safegit_dev_password@127.0.0.1:5432/safegit'

Optional: enable live Safe owner checks

When configured, SafeGit reads getOwners() and getThreshold() from the Safe and rejects non-owners.

export SAFEGIT_RPC_URL='https://ethereum-sepolia-rpc.publicnode.com'

Run migrations

Creates the repo, approval request, and signature tables.

safegit migrate

Initialize SafeGit inside your target repo

This writes .safegit.yml and registers repo → Safe mapping in Postgres.

cd /path/to/your/repo
safegit init \
  --safe 0xYourSafeAddress \
  --chain-id 11155111 \
  --threshold 2

Create an approval request for the commit

SafeGit captures the current Git metadata and creates a typed-data payload.

safegit request --ref HEAD

Owners open the approval page and sign

The page loads the backend payload, connects wallet, calls eth_signTypedData_v4, and submits the signature.

open http://127.0.0.1:8787/approve/<approvalId>

Check status

Before threshold: pending. After enough owner signatures: approved.

safegit status --ref HEAD

Enforce in CI

verify exits non-zero until the commit is approved. That is the required check.

safegit verify --ref HEAD
GitHub Actions

Turn approval into branch protection.

Add a workflow that runs safegit verify, then require the check in GitHub branch protection. This is the MVP path; a full GitHub App can create check-runs directly later.

name: Safe Verified
on: [pull_request]

jobs:
  safe-verified:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: agent-cortex/safegit@main
        with:
          database-url: ${{ secrets.SAFEGIT_DATABASE_URL }}
          ref: HEAD
